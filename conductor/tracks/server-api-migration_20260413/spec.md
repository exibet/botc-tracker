# Specification: Server API Migration

**Track ID:** server-api-migration_20260413
**Type:** Refactor
**Created:** 2026-04-13
**Status:** Draft

## Summary

Migrate all client-side Supabase queries to typed Nuxt server API routes with Zod validation, SSR auth, shared constants, and global error handling. Full plan: `docs/SERVER_API_MIGRATION.md`.

## Context

The app currently makes all Supabase queries directly from the browser via `useSupabaseClient()`. This causes: untyped join responses (`as unknown as`), anon key exposed to client, no input validation, no server-side caching, and auth-dependent UI requires `ClientOnly` wrappers with hydration flicker.

## Acceptance Criteria

- [ ] Zero `as`, `any`, `unknown` type casts in `app/` directory
- [ ] Zero hardcoded strings — all API paths, cache keys, domain enums from shared constants
- [ ] All mutations validated with Zod schemas using shared enum constants
- [ ] Global error handling via `$api` wrapper — no try/catch in composables
- [ ] Full SSR auth — server middleware resolves profile; no `ClientOnly` for auth UI, no `waitForProfile()`
- [ ] Static data (roles, stats) cached at server level
- [ ] Supabase anon key not exposed to browser (except for auth operations)
- [ ] All RLS policies remain active as security boundary
- [ ] Test coverage 80%+ across server routes, composables, and utils
- [ ] Build passes, lint passes, all tests green

## Dependencies

- `docs/SERVER_API_MIGRATION.md` — detailed migration plan with code examples
- `app/types/database.types.ts` — generated Supabase types (already exists)

## Out of Scope

- Service role endpoints (Telegram bot, cron jobs) — future work
- Supabase realtime subscriptions
- Database schema changes (migrations)
- UI/UX redesign — only removing `ClientOnly` wrappers and `waitForProfile()`

- Each phase is independently deployable — old and new approaches coexist during migration
